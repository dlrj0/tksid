### 0-1. 창 선택(포커스 필터링) 재검증 결과 — 순수 확장프로그램으로는 불가능

alt-tab/작업표시줄이 보여주는 "지금 포그라운드 창이 뭔지"는 **OS가 제공하는 정보**이고,
Chrome 확장프로그램은 샌드박스 안에 있어서 이 정보를 읽을 API가 애초에 없음. `chrome.commands`의
`global: true`는 "브라우저가 포커스를 잃어도 이 단축키는 살아있다"는 뜻일 뿐, "지금 어떤 창이
포그라운드인지"는 전혀 알려주지 않음. 그래서 "메이플이 포그라운드일 때만 반응"을 진짜로
구현하려면 OS API(예: Windows `GetForegroundWindow`)로 창 제목을 읽는 **네이티브 헬퍼
프로그램**을 Native Messaging으로 붙이는 방법뿐이고(이것도 게임에 입력을 주입하는게 아니라
읽기 전용으로 창 제목만 확인하는 거라 "감지만 함 → 매크로 아님" 판단은 동일하게 적용됨), 별도
설치가 필요해서 "네이티브 프로그램 불필요"라는 기존 결론과 다시 충돌함.

**결정: 창 필터링은 포기하고, 대신 아래 0-2로 대체.** (사용자 확인 완료)

### 0-2. 대체 설계 — 다중 단축키 + 마스터 온/오프 스위치

창을 구분하는 대신, 단축키 자체를 여러 개로 나누고 사용자가 자유롭게 재배정할 수 있게 해서
충돌 위험을 낮춤:

- 기능별로 단축키 4개를 별도 등록: **①단축키 전체 켜기/끄기(마스터 스위치) ②타이머 리셋
  ③타이머 일시정지/재개(토글) ④스누즈(다음 알람 1회 건너뛰기) 켜기/끄기**.
- 기본값은 전부 Chrome이 안전하다고 보증하는 `Ctrl+Shift+[0-9]` 범위로 지정(아래 0-3 참고).
  실제 키는 사용자가 `chrome://extensions/shortcuts`에서 원하는 대로 재배정 가능.
- 마스터 스위치를 끄면 ②③④ 세 개는 무시하지만, 마스터 스위치 자체는 항상 반응.
- 타이머 탭이 열려 있으면(볼륨 켜져 있으면) 단축키를 누를 때마다 짧은 피크음으로 피드백을
  줘서, 메이플 화면만 보고 있어도 지금 뭐가 눌렸는지 알 수 있게 함(팝업을 안 열어도 됨).

### 0-3. 중요 정정 — "충돌해도 상관없는 단축키"는 없음

Windows의 전역 단축키(`RegisterHotKey`, Chrome의 global command가 내부적으로 쓰는 방식)는
그 키 조합이 눌리는 순간 **OS가 입력을 확장프로그램 쪽으로만 통째로 가로채고, 원래 포커스된
창에는 아예 전달하지 않음.** 즉 어떤 명령을 예를 들어 `Ctrl+A`로 재배정하면 "타이머도
반응하고 다른 앱의 전체선택도 같이 동작"하는 게 아니라, **이 확장프로그램이 켜져 있는 동안은
어떤 앱에서도 `Ctrl+A`가 아예 안 먹게 됨.** 마스터 스위치를 꺼도 이 가로채기 자체는 풀리지
않음(꺼도 "반응만 안 할 뿐" 키는 여전히 삼켜짐 — Chrome 확장프로그램은 런타임에 단축키
등록을 해제하는 API가 없음). 그래서 4개 전부 `Ctrl+Shift+[0-9]` 범위처럼 평소에 잘 안 쓰는
조합을 기본값으로 유지하는 걸 권장하고, 팝업 UI에도 이 내용을 안내 문구로 넣음.
## 1. 배경 / 목표

- 사이트(MesoTimer)에서 설치기 등 쿨타임을 스크롤로 리셋해 지속시간과 싱크를 맞추며 사용 중.
- 문제: 사냥 중 마우스로 손을 옮겨야 해서 불편함 → **키보드 단축키로 리셋**하고 싶음.
- 핵심 제약: 사용자가 실제로 보고 있는 창은 **메이플 게임 창**이지, 브라우저(타이머 탭)가 아님.
  즉 단축키는 **브라우저가 포커스를 잃은 상태에서도 동작**해야 함.
- 지난 대화에서 이미 "게임에 입력을 주입하지 않고 감지만 하는 구조는 매크로/핵 정의에 해당하지
  않는다"는 정책 판단을 했음. 이번 기능도 동일한 성격(감지만 함, 게임에 아무 것도 넣지 않음)이라
  그 판단이 그대로 적용됨.

## 2. 핵심 결론 (기술 재검증 완료 — 방향이 바뀐 부분)

이전 대화에서는 "일반 확장프로그램은 전역 단축키를 못 잡으니 네이티브 헬퍼 프로그램이
필요할 것"이라고 안내했었는데, **이 부분은 정정합니다.**

Chrome 확장프로그램의 `chrome.commands` API는 커맨드를 `"global": true`로 등록하면
**브라우저가 포커스를 잃은 상태(= 메이플이 포그라운드인 상태)에서도 단축키가 동작**합니다.
별도의 네이티브 프로그램이나 로컬 서버 없이 확장프로그램만으로 해결 가능합니다.

- 근거: Chrome 공식 문서(`chrome.commands`) — "Beginning in Chrome 35, extension
  developers can optionally mark a command as global. Global commands also work
  while Chrome does not have focus."
- 단, 개발자가 manifest에 지정할 수 있는 `suggested_key`는 **global 커맨드에 한해
  `Ctrl+Shift+[0-9]` 조합으로 제한**됨 (다른 앱 단축키를 가로채는 위험을 줄이기 위한
  Chrome 자체 정책). 예: `Ctrl+Shift+9`.
- 사용자는 `chrome://extensions/shortcuts` 화면에서 이 기본값을 다른 조합으로
  **수동으로 재지정**할 수 있음. 하지만 확장프로그램 코드가 임의의 키 조합을
  프로그래밍적으로 등록하는 것은 불가능함 (아래 "제약사항" 참고).
- ChromeOS는 global 커맨드를 지원하지 않음 (참고만, 이 프로젝트엔 큰 영향 없을 가능성 높음).

## 3. 왜 스크롤 대신 이 방식인가 (대안 비교, 참고용)

| 방식 | 메이플 포커스 상태에서 동작 | 손 이동 필요 | 추가 프로그램 필요 |
|---|---|---|---|
| 마우스 스크롤 (기존) | X (타이머 탭 포커스 필요) | O (마우스로) | 없음 |
| 페이지 내 키보드 리스너 (index.html에 이미 있음, KeyR 등) | X (타이머 탭 포커스 필요) | O (탭 전환 필요) | 없음 |
| **`chrome.commands` global 단축키 (이번 목표)** | **O** | **X** | **없음** |
| OS 레벨 저수준 후킹 네이티브 앱 | O | X | 있음 (불필요하게 무거움 + 안티치트 감지 카테고리에 걸릴 여지) |

→ `chrome.commands` global이 가장 가볍고, 별도 상주 프로그램이 없어서 "메이플과 함께
실행되는 비정상 프로그램"으로 오인될 여지도 최소화됨. **이 방식으로 확정.**

## 4. 제약사항 (구현 전에 반드시 인지할 것)

1. **팝업 안에 "키를 눌러서 직접 캡처하는" 커스텀 단축키 설정 UI는 만들 수 없음.**
   `chrome.commands`는 실제 키 바인딩을 JS로 `set()`하는 API가 없음. 바인딩 변경은
   오직 브라우저 자체 화면(`chrome://extensions/shortcuts`)에서만 가능함.
   → 팝업이 할 수 있는 일은 "현재 단축키가 뭔지 보여주기"와 "설정 화면으로 이동하는
   버튼"까지임. 사용자가 "팝업창에서 단축키 설정 UI"를 요청했지만, 실제 키 입력 캡처는
   브라우저 네이티브 화면에서 이뤄진다는 걸 UI 문구로 명확히 안내해야 함(오해 방지).
2. **(확인 완료) `chrome.commands.openShortcutSettings()`는 Chrome에서 지원하지 않음** —
   Firefox 전용 API임 (MDN 확인, Chrome 쪽 공식 문서에도 이 메서드 자체가 없음). Chrome에서
   설정 화면을 여는 유일한 방법은 `chrome.tabs.create({ url: "chrome://extensions/shortcuts" })`
   이고, 구현에도 이 방식을 사용함.
3. 확장프로그램 하나가 등록 가능한 suggested 단축키는 최대 4개. 지금은 리셋 1개만 필요.
4. **메이플 클라이언트 자체 단축키와 충돌 여부는 사용자가 직접 확인해야 함.**
   `Ctrl+Shift+숫자` 조합이 메이플 기본 단축키와 겹치지 않는지 실제 게임에서 테스트 필요
   (겹치면 게임 쪽 동작이 우선될 수 있고, 반대로 확장프로그램이 게임 단축키를 먹어버릴
   수도 있음 — 실측 필요).

## 5. 구현 계획 (구현 완료 — 아래는 실제 반영된 최종 내용)

### 5-1. `manifest.json`
`commands` 키에 4개 등록 (모두 `global: true`, 기본값은 전부 `Ctrl+Shift+[0-9]` 범위 —
0-3 참고):

```json
"commands": {
  "toggle-hotkeys-master": {
    "suggested_key": { "default": "Ctrl+Shift+0" },
    "description": "단축키 전체 켜기/끄기",
    "global": true
  },
  "reset-timer": {
    "suggested_key": { "default": "Ctrl+Shift+9" },
    "description": "메이플 타이머 리셋",
    "global": true
  },
  "toggle-pause-timer": {
    "suggested_key": { "default": "Ctrl+Shift+8" },
    "description": "메이플 타이머 일시정지/재개",
    "global": true
  },
  "toggle-snooze": {
    "suggested_key": { "default": "Ctrl+Shift+7" },
    "description": "다음 알람 한 번 건너뛰기 켜기/끄기",
    "global": true
  }
}
```
- 별도 permission 불필요 (commands는 manifest key만으로 동작). suggested_key는 최대 4개까지만
  자동 지정 가능한데 정확히 4개를 다 씀 — 앞으로 커맨드를 더 추가하려면 suggested_key 없이
  정의하고 사용자가 `chrome://extensions/shortcuts`에서 수동으로 배정해야 함.

### 5-2. `background.js`
- 타이머 탭 목록(`timerTabIds`)을 `chrome.storage.session`에 저장해 추적한다.
  `watcher.js`가 로드 시 보내는 `MAPLE_TIMER_PAGE_READY` 메시지로 등록,
  `chrome.tabs.onRemoved`에서 정리. (여러 타이머 탭이 열려 있으면 **전부**에
  브릿지 메시지를 보냄 — 기존 `resolveTargetTabs`의 "선택 안 하면 전체 대상"
  패턴과 일관성 유지. 열린 결정사항이었는데 이걸로 확정.)
  - **(버그 수정, 2026-07-20)** 처음 구현 시엔 메모리상의 `Set`으로 관리했는데,
    MV3 서비스워커는 ~30초 유휴 시 종료됐다가 다음 이벤트(단축키 등)에서 다시
    깨어나며 이때 메모리가 초기화된다. 타이머 탭은 `MAPLE_TIMER_PAGE_READY`
    신호를 로드 시 **딱 한 번만** 보내므로, 사냥 중 서비스워커가 한 번이라도
    꺼졌다 깨면 재등록 기회가 없어 리셋/일시정지 단축키가 먹통이 될 수 있었음
    (실사용 패턴상 자주 발생 가능). `pausedTabIds`/`selectedTargets`와 같은
    패턴으로 `chrome.storage.session`에 저장하도록 수정해 해결. 관련 헬퍼:
    `getTimerTabIds()` / `addTimerTabId()` / `removeTimerTabId()`.
- `chrome.commands.onCommand`에서 4개 커맨드 분기:
  - `toggle-hotkeys-master`: `storage.local.hotkeysEnabled` 반전 → 배지 갱신
    (꺼지면 회색 "OFF" 배지) → 타이머 탭에 `MAPLE_TIMER_HOTKEY_MASTER_TOGGLE`(피드백음용)
    브로드캐스트. **마스터 스위치 자체는 꺼진 상태에서도 항상 반응.**
  - `reset-timer` / `toggle-pause-timer`: `hotkeysEnabled`가 꺼져 있으면 무시하고,
    켜져 있으면 각각 `MAPLE_TIMER_HOTKEY_RESET` / `MAPLE_TIMER_HOTKEY_TOGGLE_PAUSE`를
    타이머 탭에 전송.
  - `toggle-snooze`: 페이지와 무관하게 `storage.session.snoozeNextAlarm`을 바로 반전
    (기존 팝업 체크박스와 저장소를 공유하므로 서로 자동 동기화됨) → 피드백음용 메시지 전송.
- 기존 `resolveTargetTabs`/`selectedTargets`(미디어 제어 대상)와는 완전히 분리된 별도
  로직으로 구현 — 헷갈리지 않게 유지.

### 5-3. `watcher.js` (content script — 브릿지 역할)
- 로드 시(타이머 페이지 확인 후) `chrome.runtime.sendMessage({ type: "MAPLE_TIMER_PAGE_READY" })` 전송.
- `chrome.runtime.onMessage`에서 4가지 브릿지 메시지를 각각 대응하는 `CustomEvent`로
  변환해 `document`에 dispatch (`MAPLE_TIMER_HOTKEY_RESET` / `_TOGGLE_PAUSE` /
  `_MASTER_TOGGLE` / `_SNOOZE_TOGGLE`, 뒤의 둘은 `detail: { enabled }` 포함).
- PiP가 열려 있어도 문제없음: PiP는 `#app-container` DOM만 별도 창으로 옮길 뿐 페이지
  스크립트 자체(및 `document` 이벤트 대상)는 원래 탭에 그대로 있음 — `setAlarmClass`/
  `notifyReset`이 이미 이 전제로 동작 중이라 별도 PiP 분기 불필요.

### 5-4. `타이머.html` (페이지 스크립트 쪽)
기존 `keydown` 리스너 근처에 4개 이벤트 리스너 추가, **전부 기존 함수 재사용**:
```js
document.addEventListener('MAPLE_TIMER_HOTKEY_RESET', () => fastRestart());
document.addEventListener('MAPLE_TIMER_HOTKEY_TOGGLE_PAUSE', () => {
  if (state.running && !state.paused) core.pause();
  else if (state.running && state.paused) core.resume();
  else core.start();
}); // 스페이스바/클릭과 동일한 3단 토글
document.addEventListener('MAPLE_TIMER_HOTKEY_MASTER_TOGGLE', (e) => audio.playPip(e.detail?.enabled ? 750 : 300));
document.addEventListener('MAPLE_TIMER_HOTKEY_SNOOZE_TOGGLE', (e) => audio.playPip(e.detail?.enabled ? 850 : 350));
```
- `fastRestart()`/`core.pause()`/`core.resume()`/`core.start()`는 이미 내부에서
  `notifyReset()` 및 필요한 사운드(`audio.playPip`)를 호출하므로 기존
  `data-reset-seq`/`alarm-state` → `watcher.js` 감지 → 미디어 재생 재개 파이프라인이
  **그대로 자동 연결됨.**
- 마스터/스누즈는 이 페이지에 대응 함수가 없어(확장 저장소만 바뀌는 동작) 피드백용
  피크음만 재생 — 켜짐=높은 음, 꺼짐=낮은 음(기존 재개=660Hz/일시정지=220Hz 패턴과 동일한 규칙).

### 5-5. `popup.html` / `popup.js` (설정 UI)
- "전역 단축키" 섹션 추가:
  - 마스터 온/오프 체크박스(`storage.local.hotkeysEnabled`, 단축키와 저장소 공유라
    양쪽 어디서 바꾸든 자동 동기화).
  - `chrome.commands.getAll()`로 4개 커맨드의 현재 바인딩(`shortcut` 필드)을 목록으로 표시.
  - "크롬 단축키 설정 화면 열기" 버튼 → `chrome.tabs.create({ url: "chrome://extensions/shortcuts" })`
    (4번 제약사항 2번 — `openShortcutSettings()` 미지원 확인 완료).
  - 안내 문구: 팝업에서 키 캡처가 안 된다는 점 + "꺼도 키 자체는 이 확장프로그램이
    가져간다"는 0-3 내용을 사용자용으로 풀어서 안내.

## 8. UI/UX 점검 및 단축키 기능 확장 (버전 2.5)

기능이 늘어나면서(자동 on/off, 스누즈, 단축키 마스터+개별 6개, 정지된 탭 목록, 제어
대상 탭 선택까지) 팝업이 "처음 보면 뭐부터 봐야 할지 모르는" 상태가 될 위험이
커져서 점검했음. 아래는 그 점검 결과와 결정 사항.

### 8-1. 팝업 UI — 점검 결과: 정리 필요했음, `<details>` 기반으로 재구성

- **문제**: 체크박스 2개 + 단축키 안내/목록 + 정지된 탭 목록 + 탭 선택 UI(3단계:
  빠른 추가/선택됨/후보)가 전부 한 화면에 세로로 나열돼 있어서, 처음 설치한
  사람 입장에서는 "이 확장프로그램이 지금 뭘 하고 있는지"조차 스크롤을 다 해봐야
  알 수 있었음. 특히 "제어 대상 탭" 관련 3개 목록은 **선택 안 하면 전체 탭이
  기본 대상**이라 대부분의 사용자는 평생 건드릴 필요가 없는 고급 설정인데, 항상
  펼쳐진 채로 화면의 절반 이상을 차지하고 있었음.
- **결정**: 별도 라이브러리나 탭 전환 JS 없이, 네이티브 `<details>/<summary>`로
  두 그룹을 접어둠(처음 보는 사람도 "▶ 눌러서 펼치는 곳"이라는 걸 브라우저
  기본 UI로 바로 인지함):
  - `⌨️ 전역 단축키`: 열어둔 채 유지(이 확장프로그램의 핵심 기능이라 완전히
    숨기진 않되, 그 안의 "단축키 7개 목록/키 배정"은 한 번 더 접어서 평소엔
    "마스터 on/off"만 보이게 함).
  - `🎯 제어 대상 탭 (고급, 선택사항)`: 기본 닫힘. 라벨에 "고급, 선택사항"을
    명시해 안 건드려도 된다는 걸 먼저 알려줌.
  - "지금 정지된 탭" 목록은 상태를 보여주는 정보성 섹션이라 계속 항상 펼쳐둠.
  - `h1` 아래 한 줄 설명(`subtitle`) 추가 — 설명 없이 팝업만 열어도 "타이머
    알람에 맞춰 다른 탭 영상을 자동 정지/재생, 단축키로도 제어 가능"이라는
    핵심을 바로 알 수 있게.

  수정 없이 그대로 동작함.

### 8-2. 단축키 실행 피드백 — 결정: 사이트의 OS 알림 재사용 X, 오디오 핀 확장 O

`타이머.html`은 이미 `Notification` API로 "설정한 시간이 되었습니다" 같은
**타이머 알람 전용** OS 알림을 보내는 기능이 있음(`sendAlarmNotification`,
체크박스로 opt-in). 이 채널을 단축키를 누를 때마다도 재사용하는 안을 검토했는데
**기각**:
- 그 알림은 "탭을 안 보고 있어도 절대 놓치면 안 되는 이벤트(알람)"용으로 설계된
  채널이라, 리셋처럼 사냥 중 자주(분 단위로) 누르는 단축키에 그대로 쓰면 OS
  토스트가 계속 뜨면서 오히려 방해가 됨. 알림 자체에도 클릭 시 `fastRestart()`가
  걸려 있어서, 매번 뜨는 알림을 무심코 클릭했다가 의도치 않게 리셋될 위험도 있음.
- **대신** 이미 마스터/스누즈 토글에 쓰던 **피크음(`audio.playPip`) 패턴을 그대로
  확장**함(신규 3종 단축키도 동일 규칙: 켜짐/발생=높은 음, 꺼짐=낮은 음). 이 방식은
  화면을 안 보고 있어도 들리고, `state.sound`가 꺼져 있으면 자동으로 무음이 되며,
  알림창처럼 클릭 유발 위험도 없음. 정지/재생류(force-pause/force-resume)는
  배지(파란 아이콘 숫자)로도 결과가 바로 보이므로 이중으로 확인 가능.
- **"어떤 키가 언제 어떤 기능으로 동작했는지" 로그/토스트 UI는 오버엔지니어링으로
  판단해 넣지 않음.** 개인 사냥용 도구에서 실행 이력을 남겨야 할 이유가 약하고,
  피크음 톤(주파수)만으로 "뭐가 눌렸는지" 구분되는 게 이미 충분한 피드백 수준이라
  판단. 나중에 실사용해보고 "그래도 헷갈린다"는 판단이 서면 그때 짧은(1~2초)
  인페이지 토스트 정도를 `타이머.html`에 추가하는 걸 고려 — OS 알림/로그까지는
  안 가는 걸로.

### 8-3. 단축키로 새로 추가한 기능 3종 (기존 구현 기능 재사용, 새 로직 없음)

Chrome 공식 문서 기준으로 확인: **manifest의 `commands`는 개수 제한 없이 여러 개
정의 가능하고, 그중 `suggested_key`(자동 기본 배정)만 최대 4개까지 지정 가능함.**
5번째부터는 `suggested_key` 없이 정의해두면 사용자가 `chrome://extensions/shortcuts`
화면에서 원하는 것만 골라 수동으로 키를 배정할 수 있음 — "10개 넘게 있어도
되지만 사용자가 원하는 3개만 골라 쓰게" 하고 싶다는 요청에 정확히 맞는 구조.
(출처: Chrome for Developers `chrome.commands` 문서 — "An extension can have many
commands but only 4 suggested keys can be specified. The user can manually add
more shortcuts from the chrome://extensions/shortcuts dialog.")

이미 구현돼 있던 기능 중, 사냥 중 "지금 즉시" 눌러서 쓰면 실제로 도움이 될 만한
것만 추려서 3개를 단축키로 추가함(전부 기존 함수 재사용, 새 로직 없음):

| 커맨드 이름 | 기능 | 재사용한 기존 로직 |
|---|---|---|
| `force-pause-media` | 알람을 기다리지 않고 지금 즉시 대상 탭 미디어 전체 정지 | `pauseTargetTabs()`(기존 `handleAlarm` 로직을 함수로 추출해 공유) |
| `force-resume-media` | 리셋을 기다리지 않고 지금 즉시 정지시켰던 탭 재생 | `resumeTargetTabs()`(기존 `handleReset` 로직을 함수로 추출해 공유) |
| `toggle-auto-pause` | 팝업 맨 위 "자동 정지·재생" 체크박스와 동일한 기능 on/off | 기존 `storage.local.enabled` 그대로 |

- **선정 기준**: (1) 이미 background.js에 구현돼 있던 기능이라 새 위험/버그
  표면이 거의 없음. (2) "팝업을 열 필요 없이 게임 중 바로" 쓸 상황이 실제로
  있음(예: 보스 페이즈 직전에 알람을 기다리지 않고 바로 영상을 죽이고 싶다,
  반대로 알람 없이도 바로 다시 틀고 싶다, 잠깐 자동 반응 자체를 꺼두고 싶다).
  (3) 기존 4개와 성격이 겹치지 않음(기존 4개는 "타이머" 또는 "확장프로그램
  스위치류", 이 3개는 "미디어 제어"에 직접 대응).
- **채택하지 않은 후보**: "현재 활성 탭을 대상에 추가/제거하는 토글" — 단축키를
  누르는 시점엔 보통 브라우저가 아니라 메이플이 포그라운드라 "현재 활성 탭"이
  사용자가 기대하는 탭이 아닐 가능성이 높아 오히려 혼란을 줄 수 있어 보류.
  나중에 필요해지면 5-2/8-3 패턴 그대로 추가 가능(슬롯 제한 없음).
- `force-pause-media`/`force-resume-media`는 `storage.local.enabled`(자동 기능
  on/off)와 **무관하게 항상 동작**함 — 자동 반응은 꺼둬도 사용자가 지금 직접
  누른 명시적 요청은 존중한다는 원칙(8-2 배지/피크음 피드백과 함께 적용).
- 새 3개는 `suggested_key`를 지정하지 않음(위 Chrome 제약 때문에 이미 4개를
  다 씀) → 기본은 "(설정 안 됨)"이고, 필요한 사람만 팝업의 "사용 가능한 단축키
  목록 보기 / 키 배정하기"에서 원하는 키로 직접 배정.

## 10. 정책 관련 메모 (참고용, 이전 대화 요약)

- 이 기능은 게임에 어떤 입력도 주입하지 않고, Chrome이 공식 제공하는 `chrome.commands`
  API로 단축키를 감지만 함 → 매크로/핵 정의에 해당하지 않는다는 이전 판단이 그대로 유지됨.
- 오히려 별도 네이티브 후킹 프로그램이 필요 없어졌으므로, "메이플과 함께 실행되는
  비정상 프로그램" 카테고리로 오인될 리스크도 이전 설계보다 낮아짐.
- 그래도 100% 공식 확답은 아니므로, 배포/공유 규모가 커지면 넥슨 고객센터 문의를
  권장한다는 기존 결론은 유지.
